# 日课写作

日课写起来很简单，别想复杂。这个 skill 只有四件事：**校正内容 → 按需配案例 → 算编号和日期 → 我说"上传"才发星球。**

---

## 第零步：事实核查（涉及新闻/数据/公司动态必做）

日课里如果提到任何具体事实——公司上市、某人说了什么、某项数据、最新报告——**先用 `web-access` skill 查证，再动笔。**

常见需要核查的情况：
- 「XX公司上市了」「XX融资了」——查是否属实，查时间节点
- 「XX研究报告显示……」——查报告是否存在，数据是否准确
- 「马斯克/某名人说了……」——查原话，别用二手转述当事实
- 具体数字（估值、比例、成本）——核对数量级对不对

**核查结果处理：**
- 事实属实：继续写
- 事实有误或存疑：告诉我哪里不对，等我确认后再写
- 查不到：注明「待核实」，不要凭印象写

---

## 第一步：内容校正（默认行为，最重要）

日课是我用嘴说的、语音转出来的文字。**我就要用我自己的表达方式，你不要大改。**

你只做这几件事：

1. **补全跳步的逻辑** —— 我点到为止的地方补一句，省略的推导补上。比如我说"成功率下降"，补成"公众号推荐流的推流成功率下降"；我说"三头"，改成"肱三头肌"。
2. **理顺前后顺序** —— 我说话有时颠倒，帮我把因果、先后捋顺。
3. **改掉口误和模糊词** —— 错别字、明显口误、指代不清的词，换成准确的。
4. **删纯口水重复** —— "就是那个那个""然后然后"这种删掉。

**不要做的事（红线）：**

- 不要把我的大白话改成书面语，不要"润色"。
- 不要加无意义的金句、不要套华丽结尾。
- 不要改我的语气和句式特征（参考 `02-方法论/AI指令库/写作指令/写作风格.md`，但日课比公众号更口语、更随意）。

**目标风格：认知层，不是方法层。**

日课的核心竞争力是判断，不是方法。写完问自己：读者读完，认知有没有发生变化？有，才算写对了。

写法要求：
- **开头**：`今早，聊聊"XX"这个话题。` 然后直接给出核心判断或反常识观察，不铺垫不绕弯。
- **结构**：用汉字数字小标题分节（一、二、三、四），每节做一个概念辨析或判断，不是并列堆砌，是层层推进。
- **语气**：`我的体感是` `我觉得` `这里有个判断` ——有明确的个人立场，不写"大家都认为"。
- **长度**：有实质内容就写够，别刻意压短。800-1500字是正常范围。
- **出发点**：从「一个概念，大家没想清楚」出发，不是从「我今天做了什么」出发。

## 风格参考（逐字稿，写作时对照）

以下是目标风格的真实样本，写日课时对照结构和语感：

---

**样本一：聊聊"冲突"**

今早，聊聊"冲突"这个话题。

冲突，其实是真正意义上的心性的修炼场。

成长，你越来越多地消化掉"未知"，变成"已知"，让"无知"变成"知"。

回避、逃离，是把冲突和矛盾，以及它后面的"未知"，外包给时间和别人去消化，不算自己的成长。

关于跟别人的冲突，有两个要点：
1是观点，2是边界。

身份决定关系。一位校长和二位，是一种关系；朋友和恋人，是一种关系；父母和子女，是一种关系，层层相叠，相互影响。

关系背后，都存在边界。所以，与他人的冲突，也有两种：一是观点，立场的冲突；二是，身份边界冲突。

---

**样本二：关于"个人AI化"和"老板本人AI化"的一些思考**

今早，聊聊"个人AI化"和"老板本人AI化"这个话题。

当下，很多老板都在做一件事：推公司的AI化。上系统、买工具、让团队用起来、找一个"懂AI"的人来操盘。

但我的体感是，绝大多数老板漏掉了一件最关键的事——他自己。

所以，这堂日课我想掰扯清楚一件事：个人的AI化、到公司的AI化，到底是什么关系，以及为什么我认为，老板自己的AI化，是一道必答题。

一、先分清：这是两个物种的问题

关于个人怎么拥抱AI，我之前有一个框架——工具→加持→AI驱动→新物种，四个阶段。它衡量的是"我自己的产出"，被AI延伸了多少。但"老板的AI化"，不是这个框架的放大版。

因为老板手里有一样普通人没有的东西：他对一个"产出系统"负责，而非对"自己的产出"负责。

这就出现一个断层——一个老板，完全可以自己还停在"加持"阶段（亲自用AI很多），但他的公司已经是AI驱动的（他花钱雇人、搭系统把它建起来了）。

反过来也成立：一个老板自己玩得agent玩得飞起，但公司业务一点没变，离他也还是推不动。

所以"个人AI化"和"老板AI化"，既有关联，也不完全等同。

二、老板AI化的终局，不是某个场景，是一个"脱钩信号"

很多人想象老板的AI化，想象的是一个画面：满屏的agent，酷炫的工作台。其实，那不是本质。

我理解的终局，是一个"脱钩信号"——老板的产出能力，能管多少人、能打多少事、能投多少时间。

---

**样本三：制造AI化场地的一些思考**

高效，是工作上面一套围绕效率的整体考量，不是一套工具。

很多老板想象老板的AI化，想象的是一个画面：满屏的agent，酷炫的工作台。

但我的体感是，绝大多数老板漏掉了一件最关键的事——他自己。一个人只有自己拥有了AI的能力，他才不可能对AI不够了解。有些事情，AI可以比你更懂，但不应该比你更懂你的业务。

第一，帮你解决问题的人，是一个native视角的人。

一个能把AI更好地渗透进业务中的人，正是因为他在这个领域同时懂AI和懂业务，他才能把两者很好地融合。你不能指望一个不懂业务的人，把AI用于你的业务；也不能指望一个只懂业务的人，知道怎么用AI让业务更好。

---

一句话：**你是帮我把判断说清楚，不是帮我描述我做了什么。** 写完自问：读者认知有没有升级？

---

## 写完必做：两个自查

**自查一：读者视角扫描**
逐句问：一个不了解背景的人看得懂吗？
- 有没有专有名词没解释？（比如AWS、OKR、ROI……出现就顺手一句话解释）
- 有没有默认读者知道的背景信息，其实他们不知道？

**自查二：对比判断检查**
凡是用了"不是A，是B"这个句式，写完必须确认：A和B真的是非此即彼吗？
- 如果两个都重要、都成立，改成"A在打，B也在打"
- "不是X"这种说法说得过头，就是在给读者挑错机会

---

## 第二步：案例（仅按需，默认不配）

**默认不配案例。** 日课的观点是我新鲜的第一手想法，本身就够。

只有当我**明确说**"配个案例 / 加个例子 / 这里举个真实案例"时，你才去搜：

1. 直接精读 `01-素材/案例库/` 里命中的文件（重点看 `学员案例库.md`）。
2. 挑一个最贴合的，避开近期日课已经用过的（扫一眼 `03-产出/日课/` 最近几篇）。
3. 案例融进我的话里，不要单独拎成一段"案例展示"。

日课观点是新内容，**不需要**回存素材库。

## 第三步：编号与日期

日课编号**按日期跳动，不是在上一篇编号上+1**。

规则：找编号最大的那篇作为锚点，取它的编号N和日期D，今天编号 = N +（今天 - D的天数差）。没写的那天，号自动跳过，所以编号里有缺口是正常的。

**错误示范**：上一篇是450，今天直接写451——这是流水号逻辑，是错的。
**正确示范**：上一篇450是6月11日，今天6月14日，差3天，今天是453。

取号别手算，跑脚本：

```bash
python3 "$CLAUDE_SKILL_DIR/rike_number.py" --keyword 关键词
```

脚本逻辑：扫日课目录找编号最大的那篇当锚点，今天编号 = 锚点编号 + (今天 − 锚点日期) 的天数差。同一天再写就 +1。

- **开头格式**：`#日课 今天是YYYY-M-D，日课第X篇，关键词：XX。`
- **文件名**：`日课X-YYYYMMDD-关键词.md`，存到 `03-产出/日课/`
- **顶部 frontmatter**：

  ```
  > 发布渠道：知识星球
  > 状态：草稿
  > 日期：YYYY-MM-DD
  ```

---

## 第四步：上传（我说"上传"才执行）

用户说"上传"，发到知识星球。

**发布内容**：去掉顶部 frontmatter（发布渠道/状态/日期那几行），只发 `#日课…` 开头的正文。

**发布方式按运行环境选：**

- **在 Cowork 里（默认）**：用 **Claude-in-Chrome 扩展**驱动我已登录的 Chrome 直接发，不用切到 Mac。步骤：
  1. `list_connected_browsers` → `select_browser` 选中我的浏览器（没连上就提示我连扩展）。
  2. `tabs_context_mcp`（createIfEmpty）开标签 → `navigate` 到 `https://wx.zsxq.com`，进入你的知识星球。
  3. 点顶部「点击发表主题…」发布框 → 点进输入区 → `type` 正文（含 `#日课…` 开头那行，段落间空行保留）。
  4. 截图核对内容无误 → 点右下「发布」。
  5. 发完截图确认，告诉我发在哪条（星球会先显示"审核中"，属正常自动审核）。

- **在我 Mac 上的 Claude Code 里**：走 `web-access` skill 的 CDP 模式 —— 连本机 `127.0.0.1:9222`（先在 `chrome://inspect` 开 remote debugging），同样打开 `wx.zsxq.com` → 进星球 → 新建动态 → 粘贴正文 → 发布。

> 说明：CLAUDE.md 里"网页一律走 web-access、别用 claude-in-chrome"那条，是针对 Mac 上 web-access 环境写的。Cowork 沙箱连不到我 Mac 的 Chrome（127.0.0.1 是沙箱自己，跨主机被白名单挡掉），所以**在 Cowork 里发日课这件事，允许且应当用 claude-in-chrome**。
